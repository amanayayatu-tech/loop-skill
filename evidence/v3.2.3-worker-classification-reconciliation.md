# v3.2.3 Worker classification reconciliation

Date: 2026-07-14

## Incident

A real paused Loop closed a Worker dispatch after the installed payload verifier
returned `DISPATCH_VALIDATION_MATRIX_MISMATCH`. The target formal report stated
`status=BLOCKED` and `execution_started=false`; no product file or provider was
touched. The small `FORMAL_REPORT_STAGED` result omitted the optional execution
classification fields, so ACK defaulted the canonical attempt to
`execution_started=true`.

This was a runtime accounting defect. It incorrectly consumed a repair slot
even though the immutable archived report proved that execution never started.

## Fix boundary

- Report staging binds a Worker report's top-level `execution_started` and
  approved `blocker_code` into the ACK-ready result.
- `DISPATCH_VALIDATION_MATRIX_MISMATCH` is an explicit deterministic
  zero-execution blocker.
- `RECONCILE_WORKER_EXECUTION_CLASSIFICATION` may repair an already archived
  misclassification only at canonical `PAUSED_AT_SAFE_POINT` with no lease or
  active outbox. Runtime re-reads the exact canonical report path, verifies its
  digest and identities, and changes only the existing attempt/latest-worker
  classification.
- Attempt history, report bytes, repair counters, Controller Pack identity,
  task identities, and provider receipts remain immutable.

## Evidence ceiling

The regression proves repository runtime accounting and a bounded canonical
reconciliation path. It does not convert the failed Local Verification into
PASS, authorize another provider call, or claim that Codex app-server process
cleanup is fixed. No raw provider response, prompt, credential, key, personal
data, database, or generated project Pack is included.
